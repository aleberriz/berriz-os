# Voice Guide

How Alejandro Berrizbeitia writes. Apply this when drafting any public-facing text:
website copy, newsletter issues, GitHub READMEs, LinkedIn posts, PR descriptions
intended for public consumption, or any AI-authored content on Alejandro's behalf.

---

## The governing principle

Copy should earn the reader's attention, not assume it.

Every sentence should do one job: give the reader something they did not have before —
a fact, a reframe, a useful distinction, or occasionally a laugh. Sentences that merely
signal effort, enthusiasm, or presence should be cut.

---

## Tone

**Precise, not dense.** Use the exact word. Avoid approximations and hedges where the
specific term exists ("causal" not "related to causality"; "squash merge" not "combine
the commits somehow").

**Plain, not simple.** The Economist writes about complex things in plain English.
That is the target. Never dumb down the subject matter, but never use jargon where a
plain word works equally well.

**Confident, not loud.** State things directly. "The semantic layer is not a BI tool
feature" is better than "Many people think of the semantic layer as a BI tool feature,
but it's actually much more than that." Cut the wind-up.

**Occasional dry wit.** Earned through precision, not decoration. A well-placed
observation that is slightly funnier than expected lands better than a joke. Never
explain the joke.

**No hype.** "Revolutionary", "game-changing", "powerful", "excited to share" — all cut.
The work speaks; the copy describes the work.

---

## Structural rules

- One idea per sentence.
- Short paragraphs (2-4 sentences for body; 1 for impact).
- BLUF structure for any piece of writing longer than a paragraph: the conclusion comes
  first, the reasoning follows.
- No bullet-point padding. If a list has fewer than three items, fold it into prose.
  If a list has more than seven items, it needs subcategories or should be a table.
- Verbs do the work. "I build" is better than "I am focused on building". "This metric
  measures" is better than "This metric is used to measure".

---

## Register by context

| Context | Register |
|---------|----------|
| Newsletter | First-person, direct, conversational but not chatty. Like a thoughtful memo from a colleague you respect. |
| GitHub README | Third-person or second-person ("this repo", "you'll find"). Functional. Not dry. |
| LinkedIn | Same as newsletter but aware that most readers will see only the first two lines. Lead with the point. |
| Website hero | 80 words or fewer. First-person. One claim, one credential, one invitation. |
| PR descriptions | Second-person ("this PR adds", "reviewers will find"). Context-heavy. The diff shows what; the description explains why. |

---

## What to avoid

| Avoid | Because |
|-------|---------|
| "I'm excited to share…" | Filler; every post would say this |
| "In today's fast-paced world…" | Throat-clearing; cut |
| "Leveraging" | Use "using" |
| "Utilize" | Use "use" |
| "Synergy", "ecosystem" (as filler) | Vague; be specific |
| Passive voice without reason | Obscures who did what |
| Emoji in long-form | Distracting; reserve for single-line social posts only |
| Oxford comma omission | Always use the Oxford comma |

## References

The writing voice is shaped by:
- *The Economist* style guide — precision, plain English, the active voice
- The Economist Education [Data Storytelling](https://education.economist.com/courses/datastorytelling)
  and [Business Writing](https://education.economist.com/courses/professionalcommunication) courses
- George Orwell's six rules (*Politics and the English Language*, 1946)

Orwell's rules, restated for this context:
1. Never use a long word where a short one will do.
2. If it is possible to cut a word, cut it.
3. Never use the passive where you can use the active.
4. Never use jargon if you can think of an everyday equivalent.
5. Break any of these rules rather than write something outright barbarous.
